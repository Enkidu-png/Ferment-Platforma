<remaining_work>

## Architectural migration still needed — requires a NEW PHASE:

The user's goal: "remove Payload everywhere where Supabase can take over — no Payload CMS
users in the end, everything on Supabase."

### Critical architectural issue discovered:
Two conflicting tenant data stores exist:
- `src/app/auth/confirm/route.ts` creates tenants in **Supabase** (`supabase.from("tenants")`)
  and links via `user_tenants` join table
- tRPC procedures (`tenants.getOne`, `checkout.verify`, `products.getMany`) query tenants
  from **Payload CMS** (`ctx.db.find({ collection: "tenants" })`)
- → New Supabase-registered users have tenants in Supabase ONLY — Payload tenant queries
  return nothing for them. The tenant storefront is broken for all new users.

### Work items for next phase:

1. **`checkout.verify`** — Remove Payload user lookup (`ctx.db.findByID({ collection: "users" })`).
   Instead query Supabase `user_tenants` to find the user's tenant, then query Payload tenant
   by slug (or Supabase tenant by id) for `stripeAccountId`.

2. **`tenants.getOne` tRPC** — Decide: query Supabase `tenants` table or Payload `tenants`
   collection? Currently Payload; but data is only in Supabase for new users.

3. **`Orders.user` field** — Change from `relationship` (pointing at Payload users collection)
   to `text` (stores Supabase UUID directly). Migration required.

4. **`Reviews.user` field** — Same change.

5. **Stripe webhook** (`src/app/(app)/api/stripe/webhooks/route.ts`) — Removes
   `payload.findByID({ collection: "users" })` lookup. Use `data.metadata.userId`
   (already a Supabase UUID) directly as the `user` text field in the order.

6. **`account.updated` webhook** — Currently updates Payload `tenants`. Decide: keep in
   Payload (for product scoping via multi-tenant plugin) or move to Supabase.

7. **Payload `Users` collection** — User wants no Payload users at all. Payload admin needs
   some user collection to function. Decision needed:
   A) Keep minimal `Admins` collection (CMS super-admins only, separate from app users)
   B) Remove Payload admin entirely (Payload becomes pure headless API)
   USER CONFIRMED: option A or B — awaiting decision when resuming.

8. **`auth/confirm` route** — Should ALSO create a Payload tenant record (with a
   `supabaseUserId` text field) so that Payload's multi-tenant plugin can scope products.
   OR move product-to-tenant scoping out of Payload entirely.

9. **`src/lib/access.ts`** — `isSuperAdmin` uses Payload user `roles` field. Without Payload
   users this breaks. Needs Supabase-based role check (e.g. custom JWT claim or Supabase
   `user_roles` table).

10. **`checkout.purchase`** — `ctx.user.email` is now set correctly, but `userId` in Stripe
    metadata is Supabase UUID — already correct after session's fixes.

<decisions_made>

  from app user auth. User wants this resolved (Supabase-only) but decision on HOW is pending.
